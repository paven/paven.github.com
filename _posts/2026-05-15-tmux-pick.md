I have a complicated relationship with terminal setup. I spend time on it, it drifts, I ignore it, repeat. Most improvements disappear the moment the novelty wears off.

tmux-pick did not disappear.

## The problem

I use Kitty and tmux. Every new terminal window was a small decision: attach to something existing, or start fresh. The problem was that existing sessions were invisible. Out of sight, out of mind - they piled up forgotten in the background, or worse, I started duplicate sessions for work I had already begun.

What I wanted: open a terminal, get a list, pick.

## The solution

A small bash script that runs on startup. If tmux has sessions, fzf shows them. You pick one, you are in. If there are none, a new session starts automatically.

```bash
CHOICE=$(build_list | fzf \
  --prompt="tmux> " \
  --height=40% \
  --reverse \
  --header="enter=attach  d=delete  r=refresh  esc=plain shell")
```

Four keys. Enter attaches. d deletes a session and refreshes the list in place. r refreshes. Esc drops you into a plain shell.

## The bug I did not see coming

Esc was supposed to exit to a plain terminal. It did not work.

The reason: the .zshrc condition that launches tmux-pick checks for Kitty and no active tmux session. Esc exited the picker, started a new zsh, which sourced .zshrc, which saw Kitty and no tmux, and launched the picker again. Infinite loop.

The fix was one environment variable:

```bash
# In tmux-pick, on Esc:
exec env KITTY_PLAIN_SHELL=1 "${SHELL:-zsh}"

# In .zshrc:
if [[ -n "$KITTY_WINDOW_ID" && -z "$TMUX" && -z "$KITTY_PLAIN_SHELL" ]]; then
  exec tmux-pick
fi
```

Small bug, obvious in hindsight. The kind of thing that only surfaces when you actually use the tool.
